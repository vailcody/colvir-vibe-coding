## Блок 2: Dify — создание воркфлоу

### 0:08–0:13 — Что такое Dify и зачем (5 мин)

**Что говорить:**

- «Зачем Dify, а не просто вызвать OpenAI API из кода? Три причины:»
  1. «Визуально видно весь пайплайн — легко дебажить и менять»
  2. «Можно менять промпты без деплоя кода»
  3. «Есть готовый API — не надо писать обёртки»
- «Для а это особенно актуально: аналитики могут настраивать промпты, не трогая код»

- *«Какие модели поддерживает?»* → «Все основные: OpenAI, Anthropic, локальные через Ollama. Можно даже свою модель подключить»

### 0:13–0:22 — Создание воркфлоу в Dify (9 мин)

**Что говорить (по шагам, каждый шаг сопровождать действием на экране):**

1. «Создаём новый Workflow в Studio» *(клик → New App → Workflow)*
2. «Наш пайплайн простой: файл на вход → парсинг текста → LLM → JSON на выходе»
3. «Начинаем с входной ноды. Тип — File Upload. Принимаем .txt, .md, .docx — типичные форматы спецификаций»
4. «Следующая нода — Document Extractor. Она вытаскивает текст из файла. Даже из PDF»
5. «Теперь главное — LLM-нода. Подключаем GPT-4o (или Claude). Пишем промпт:»

```
Ты — QA-инженер. На вход получаешь спецификацию программного продукта.
Сгенерируй тест-кейсы в формате JSON.

Каждый тест-кейс должен содержать:
- id (string)
- title (string) 
- description (string)
- steps (array of strings)
- expected_result (string)
- priority (high/medium/low)

Спецификация:
{{text}}

Верни ТОЛЬКО валидный JSON-массив, без markdown-обёртки.
```

6. «Выходная нода — End. Отдаём результат LLM как есть»

**Что говорить:**

**Ask (Cmd+L):**

**Agent (Cmd+I → Agent):**
- «Режим "сделай за меня". AI читает файлы, создаёт новые, пишет код, запускает команды. Это основной режим вайб-кодинга»

**Edit (Cmd+K):**
- «Режим точечной правки. Выделяете кусок кода → описываете изменение → AI меняет только этот фрагмент»
- *(Демо: выделяем print → Cmd+K → "Замени на логирование через logging")*

- «Для нашей задачи будем в основном использовать Agent — он будет собирать приложение по нашим командам»

## Блок 6: MCP

### 1:33–1:36 — Что такое MCP (3 мин)

**Что говорить:**
- «MCP — Model Context Protocol. Звучит сложно, но идея простая»
- «Представьте: Cursor умеет писать код. А MCP даёт ему руки — подключаться к базам данных, API, файловым системам, браузерам»
- «Без MCP: вы копируете данные из базы → вставляете в чат → AI анализирует»
- «С MCP: AI сам подключается к базе, сам делает запрос, сам анализирует. Вы только просите»
- «Аналогия: MCP — это USB-порт для AI. Подключил плагин — получил новую способность»

**На экране:** Схема: Cursor ↔ MCP Server ↔ [Database / API / Browser / File System]

### 1:36–1:40 — Подключение MCP-сервера (4 мин)

**Что говорить:**
- «Подключим MCP-сервер для файловой системы — самый простой пример»
- *(Открываем Cursor Settings → MCP → Add Server)*
- «Конфигурация — JSON. Указываем тип (stdio), команду, аргументы»

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/project"]
    }
  }
}
```

- «Теперь Cursor видит наш проект не просто как файлы, а как структурированные данные»

**На экране:** Cursor Settings → MCP configuration

### 1:40–1:43 — Примеры MCP для а (3 мин)

**Что говорить:**
- «Какие MCP-серверы полезны для вашей работы:»
  1. **PostgreSQL MCP** — AI читает и анализирует вашу базу. «Покажи все таблицы, связанные с модулем авторизации»
  2. **Jira/Linear MCP** — AI видит задачи. «Создай тест-кейсы для всех багов в текущем спринте»
  3. **Browser MCP (Playwright)** — AI управляет браузером. Автотесты UI без Selenium
  4. **Git MCP** — AI анализирует историю. «Какие файлы чаще всего ломаются?»
- «MCP — это мост между AI и вашей инфраструктурой. Для банковского софта — мощнейшая штука»

### 1:43–1:48 — Rules и контекст (5 мин)

**Cursor Rules (.cursor/rules):**

- «Это как .editorconfig, но для AI. Экономит тонну времени на объяснениях»

**На экране:** Создание файла .cursor/rules, демо как AI следует правилам

### 1:48–1:52 — @-mentions и контекст (4 мин)

**Что говорить:**
- «@-mentions — способ дать AI точный контекст»
- `@file` — конкретный файл: «@package.json — какие зависимости устарели?»
- `@folder` — вся папка: «@src/api — отрефактори обработку ошибок»
- `@web` — поиск в интернете: «@web — как настроить Prisma с SQLite?»
- `@docs` — документация: подключаете URL доки, AI ищет по ней
- `@codebase` — весь проект: «@codebase — где используется функция calculateRisk?»
- «Золотое правило: чем точнее контекст, тем лучше результат. @file лучше @codebase для конкретных задач»

**На экране:** Примеры @-mentions в чате Cursor

### 1:52–1:55 — Экономия токенов и большие кодовые базы (3 мин)

**Что говорить:**
- «Для больших проектов — а у а наверняка немаленькие репозитории:»
  1. **`.cursorignore`** — исключаем node_modules, билды, вендоров. Как .gitignore, но для AI
  2. **Точечные @-mentions** вместо @codebase — экономия токенов в 10-100 раз
  3. **Декомпозиция задач** — не «перепиши весь модуль», а «перепиши функцию X в файле Y»
  4. **Новый чат для новой задачи** — контекст не засоряется предыдущими разговорами
- «Практическое правило: один чат = одна задача. Как git branch — по фиче»

**На экране:** Пример .cursorignore, демо разницы в скорости ответа

**Возможные вопросы:**
- *«Можно ли использовать с монорепо?»* → «Да, но важно правильно настроить .cursorignore и использовать @folder для конкретного пакета»
- *«Как с приватным кодом?»* → «Business-план + Privacy Mode. Или self-hosted модели через MCP/Ollama»
