# Cursor IDE для опытных разработчиков — Ресёрч для воркшопа Колвир

> Контекст: B2B-воркшоп, ~20 человек, 70-80% разработчики банковского софта (Колвир). Цель — ускорить работу с помощью AI-инструментов.

---

## 1. Режимы работы Cursor

### Обзор режимов

| Режим | Назначение | Что может | Инструменты |
|-------|-----------|-----------|-------------|
| **Agent** | Построить фичу, пофиксить баг, рефакторинг | Автономное исследование кода, multi-file edits, запуск команд в терминале | Все инструменты |
| **Ask** | Понять код, спланировать работу, исследовать | Read-only поиск по кодовой базе и объяснения | Только поиск |
| **Edit** (Cmd+K) | Точечные правки в конкретном месте | Inline-редактирование выделенного кода | Контекст выделения |
| **Plan** (Shift+Tab) | Декомпозиция сложных фич | Создаёт детальный план реализации, задаёт уточняющие вопросы | Все инструменты |

### Когда какой использовать — фреймворк для опытного разработчика

**🔍 Ask Mode → Планирование и исследование**
- Используй ПЕРЕД любым сложным изменением
- Официальная рекомендация Cursor: «Планируй в Ask, реализуй в Agent»
- Идеален для impact analysis: «Если я изменю интерфейс User, что сломается?»
- Ask + хороший вопрос = 10x быстрее чем grep + ручное чтение
- **Пример промпта:**
  ```
  Мне нужно добавить multi-tenancy. Перед началом:
  1. Как структурирована текущая система аутентификации?
  2. Какие таблицы не имеют tenant_id?
  3. Какие singleton-сервисы нужно сделать tenant-aware?
  Ищи по кодовой базе тщательно. Мне нужна конкретика.
  ```

**🤖 Agent Mode → Реализация**
- Когда знаешь, что хочешь построить, и scope = 3-10 файлов
- Рефакторинг интерфейса + обновление всех потребителей
- Генерация тестов для существующего кода
- **Ключевой инсайт:** Agent с включённым auto-run (YOLO mode) = драматически более полезен. Он пишет код → запускает тесты → видит ошибки → итерирует без вашего участия
- **Когда НЕ работает:** >15-20 файлов (разбивай на подзадачи), непонятная бизнес-логика, незнакомый codebase

**✏️ Edit Mode (Cmd+K) → Точечные правки**
- Выделил код → Cmd+K → описал изменение
- Идеален для: рефакторинг одной функции, добавление типизации, изменение сигнатуры
- Не трогает другие файлы — предсказуемо и безопасно

**📋 Plan Mode (Shift+Tab) → Комплексные фичи**
- Самый недооценённый режим
- Исследует кодовую базу, задаёт вопросы, создаёт план
- План можно отредактировать до реализации
- Для задач на 20+ файлов: Plan → Review → Agent

### Золотой воркфлоу: цепочка режимов

```
Ask (исследование) → Plan (декомпозиция) → Agent (реализация) → Ask (проверка)
```

> «Ваш ПМ скинул тикет на API + 3 компонента + миграция + тесты. Вы открыли Agent, промптнули — и через 90 секунд смотрите на diff из 14 файлов, половина из которых неправильные. Проблема не в промпте. Проблема в том, что надо было начать с Plan, потом Ask для исследования, и только потом Agent.» — Developer Toolkit

## 2. Фишки Cursor для продвинутых

### 2.1 @-mentions — управление контекстом

Контекст = главная валюта при работе с AI. @-mentions позволяют дать AI ровно столько информации, сколько нужно.

| Упоминание | Что делает | Когда использовать |
|-----------|-----------|-------------------|
| `@filename` | Подключает конкретный файл | Точные правки, ссылки на паттерны |
| `@folder` | Подключает всю папку | Работа с модулем целиком |
| `@codebase` | Поиск по всей индексированной базе | Широкие вопросы, поиск зависимостей |
| `@docs` | Подключает документацию фреймворка | Вопросы по API, совместимость версий |
| `@web` | Поиск в интернете | Актуальная информация, changelog, баги |
| `@git` | История коммитов | Понять почему что-то изменилось |

**Продвинутые паттерны:**
- Комбинируй: `@auth.ts реализуй такой же паттерн для @user-service.ts`
- Не кидай `@codebase` на каждый запрос — это сжирает контекст
- Начинай с узкого контекста, расширяй только если AI говорит «мне нужно больше»

### 2.2 .cursor/rules — как «обучить» Cursor под проект

**Что это:** Файлы в `.cursor/rules/*.mdc`, которые инжектируются в каждый разговор с AI. По сути — system prompt для вашего проекта.

**Три уровня правил:**
1. **Model layer** — LLM (Claude Sonnet, GPT-4.1, etc.)
2. **Cursor layer** — встроенные промпты и инструменты Cursor
3. **Custom rules layer** — ваши `.cursor/rules/*.mdc` файлы ← **это вы контролируете**

**Пример боевого файла (Elementor Engineering):**

```markdown
---
alwaysApply: true
---
# STRICT RULES

## CRITICAL PARTNER MINDSET
Не соглашайся автоматически с моими утверждениями. Ставь под 
сомнение предположения, предлагай контраргументы.

## EXECUTION SEQUENCE (отвечай "Applying rules X,Y,Z")
1. SEARCH FIRST — ищи похожую функциональность в кодовой базе
2. REUSE FIRST — проверяй существующие паттерны, расширяй перед созданием нового
3. NO ASSUMPTIONS — используй только прочитанные файлы и сообщения
4. CHALLENGE IDEAS — если видишь проблемы или лучший подход, скажи прямо

## CODING STANDARDS
- Планируй перед кодированием
- Без комментариев в коде — пиши самодокументирующийся код
- Импорты — в алфавитном порядке
- SOLID, но без оверинжиниринга
- Файлы до 300 строк
```

**Важные нюансы:**
- Правила написаны в «martial arts» стиле — коротко, чётко, без воды. Каждое слово = токены
- Файл в `alwaysApply: true` загружается в КАЖДЫЙ разговор
- Агент «забывает» правила через несколько сообщений → периодически пиши «прочитай правила заново»
- Трюк: попроси агента отвечать «Applying rules 1,2,3…» — видно, когда он следует, а когда нет
- Когда правила в репозитории → вся команда получает единые стандарты

**Что писать в правилах — два направления:**

*Что НЕ делать:*
- Не пиши комментарии (если стандарт = без комментариев)
- Не запускай `npm run` если hot reload активен
- Не пиши документацию без запроса

*Как делать правильно:*
- Как получить данные из GitHub branch
- Как запускать тесты в нашем проекте
- Какую структуру файлов использовать

### 2.3 Работа с большими кодовыми базами

**Проблема:** Проект > 200k строк → индексация 20+ минут, Agent тянет устаревший код, Tab-подсказки из deprecated-модулей.

**Решения:**

**1. `.cursorignore` — настрой ДО первого открытия:**
```
# Артефакты сборки
dist/
build/
.next/
coverage/

# Зависимости
node_modules/
vendor/

# Генерированный код
*.generated.ts
prisma/generated/

# Большие файлы
*.wasm
*.map
*.min.js
package-lock.json
pnpm-lock.yaml
```

> ⚠️ НЕ исключай свой source code (src/, lib/, app/), тесты и конфиги — AI нужно это видеть

**2. Открывай конкретные пакеты, а не root монорепо:**
```bash
# Вместо:
cursor ~/big-monorepo

# Открывай нужные пакеты:
cursor ~/big-monorepo/packages/web \
       ~/big-monorepo/packages/api \
       ~/big-monorepo/packages/shared
```

**3. Думай «бюджетами контекста»:**
- ~200k токенов на разговор (зависит от модели)
- Типичный файл 200 строк ≈ 1000-2000 токенов
- `@src/` на 500 файлов = весь бюджет сожран
- **Правило:** начинай с минимума файлов, расширяй по запросу

**4. Summary-файлы для крупных модулей:**
- Создай `README.md` в ключевых модулях с описанием архитектуры
- AI будет использовать их вместо чтения всех файлов

**Ориентиры по здоровью индексации:**

| Размер проекта | Время индексации | Размер индекса |
|---------------|-----------------|----------------|
| <10k файлов | 1-3 мин | <500 MB |
| 10k-50k файлов | 3-10 мин | 500 MB - 2 GB |
| 50k-200k файлов | 10-30 мин | 2-5 GB |
| >200k файлов | Нужны агрессивные исключения | Цель: <5 GB |

### 2.4 Экономия токенов — какие модели для чего

**Система кредитов Cursor (с июня 2025):**
- Каждый платный план = пул кредитов (= цена плана в $)
- Разные модели тратят разное количество кредитов
- Output-токены стоят в 2-5x дороже input-токенов

**Тир-лист моделей по задачам:**

| Задача | Рекомендуемая модель | Почему |
|--------|---------------------|--------|
| Сложная архитектура, рефакторинг | Claude 4 Sonnet, Claude 3.7 Sonnet | Лучшее понимание кода и инструкций |
| Агентский режим (heavy lifting) | Claude 4 Sonnet, o3 | Отлично работают с правилами |
| Рутинные правки, boilerplate | GPT-4.1, Gemini | 2.4x больше запросов за те же деньги |
| Простой конфиг, базовая функция | cursor-small, GPT-4.1-mini | Самые дешёвые |
| Код-ревью | Claude 4 Sonnet | Лучшее критическое мышление |

**Конкретные цифры (на $20 кредитов):**
- Claude Sonnet: ~225 запросов
- Gemini: ~550 запросов (2.4x дешевле!)
- Claude Opus: ~27 запросов (дорого, для критичных задач)

**Стратегия:** Gemini/GPT-4.1 для рутины → Claude Sonnet для сложного → Opus только когда Stakes высокие

### 2.5 Multi-file editing и Agent mode

**Agent Mode + YOLO (auto-run):**
- В настройках Cursor включи auto-run с whitelist команд
- Разреши: тесты (vitest, jest, npm test), сборку (tsc, npm build), lint (eslint, prettier)
- Запрети: `rm -rf`, `git push --force`, `DROP TABLE`
- **Результат:** Agent пишет код → запускает тесты → видит ошибки → исправляет → без вашего участия

**Test-Driven Agent (лучший паттерн):**
```
Реализуй rate limiter middleware для Express API.

1. Сначала напиши тесты в src/middleware/__tests__/rate-limiter.test.ts:
   - per-IP limiting, sliding window, кастомные лимиты по роутам, bypass для healthcheck
2. Реализуй middleware в src/middleware/rate-limiter.ts
3. Запусти тесты и исправь ошибки
4. Повторяй пока все тесты не пройдут

Следуй паттернам из @src/middleware/auth.ts
```

## 3. MCP (Model Context Protocol)

### Что это

**MCP = «USB-C для AI»** — универсальный протокол, который позволяет AI-ассистенту подключаться к внешним инструментам и данным через стандартизированные API.

**Без MCP:** переключаешься между вкладками (GitHub, БД, документация, браузер)
**С MCP:** всё доступно прямо из Cursor через AI-промпты

**Техническая суть:**
- JSON-RPC 2.0 для обмена сообщениями
- Транспорт: stdio (локально) или HTTP + SSE (удалённо)
- Стандартные форматы для запросов, ответов, ошибок

### Архитектура

```
┌────────────────┐     ┌─────────────────┐     ┌────────────────┐
│  MCP Client    │────▶│  MCP Server     │────▶│  Ресурс        │
│  (Cursor IDE)  │     │  (программа)    │     │  (БД, Git,     │
│                │     │                 │     │   API, файлы)  │
└────────────────┘     └─────────────────┘     └────────────────┘
```

### Настройка в Cursor

Файл `.cursor/mcp.json` в проекте:
```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/path/to/project"]
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://user:pass@localhost:5432/mydb"
      }
    }
  }
}
```

### Топ MCP-серверов для разработчиков

| MCP Server | Что даёт | Применение для Колвир |
|-----------|---------|----------------------|
| **PostgreSQL** | SQL-запросы к БД прямо из Cursor | Анализ данных, отладка запросов, генерация миграций |
| **Git** | История коммитов, diff, blame | Понимание «почему так написано», code review |
| **GitHub** | Issues, PRs, CI/CD | Работа с задачами без выхода из IDE |
| **Filesystem** | Чтение/запись файлов | Работа с конфигами, логами |
| **Browser/Playwright** | Автоматизация браузера | E2E тестирование, скрапинг документации |
| **Docker** | Управление контейнерами | DevOps-задачи из IDE |
| **Sentry** | Ошибки и стектрейсы | Диагностика production-багов |
| **Figma** | Дизайн-токены, компоненты | Для фронтенд-разработки |

**Обновление Jan 2026:** Cursor улучшил работу с множественными MCP-серверами — динамическое управление контекстом, снижение потребления токенов на 46.9%.

### Почему MCP важен для enterprise-разработчиков

- **Банковский контекст (Колвир):** MCP к PostgreSQL = AI может анализировать схему БД, генерировать оптимизированные запросы, находить проблемы в данных — всё из IDE
- **Безопасность:** MCP-серверы работают локально, данные не уходят наружу (в отличие от копирования в ChatGPT)
- **Масштабируемость:** Один раз настроил `.cursor/mcp.json` → вся команда получает одинаковый tooling

## 4. Стратегии работы с ошибками AI-агентов

### Почему AI-агенты зацикливаются

**Корневая причина:** AI не хранит «живое» понимание кодовой базы. Он работает с текстом в контекстном окне. После 15 сообщений все неудачные попытки, «подожди, откати это», полу-применённые патчи — всё это сидит в контексте. На 15-м сообщении агент не решает текущую проблему — он пытается примирить противоречивое состояние из сообщений 3, 7 и 12.

**Цикл fix→break→fix→revert:**
1. Agent фиксит баг
2. Фикс ломает тест
3. Просишь пофиксить тест
4. Agent фиксит тест... откатывая оригинальный фикс
5. Возвращаешься в начало

> «В момент, когда агент ссылается на ошибку, которую вы уже исправили два сообщения назад — контекст скомпрометирован. Остановите сессию.» — сообщество опытных вайб-кодеров

### Multi-file = экспоненциально хуже

- Просишь «пофикси auth flow» без указания файла → агент трогает middleware, роуты, session config, тесты за один проход
- Каждое изменение = downstream-эффекты, которые модель не отслеживает
- **Практический потолок:** 1500-3000 строк активного кода в scope на одну сессию

### Стратегии спасения

#### 1. Правило 8 сообщений
Если агент зациклился и глубина разговора >8 сообщений — **закрой чат, открой новый**. Не пытайся «дообъяснить». Вставь текущую ошибку + только релевантный файл в чистую сессию — качество ответа будет драматически лучше.

#### 2. Git Checkpoint Workflow (must have!)
```bash
# Перед КАЖДЫМ AI-промптом, который изменит код:
git add -p
git commit -m "checkpoint: before AI refactor"

# После успешного AI-изменения:
npm test
git commit -am "feat: password reset wired to /api/auth/reset"

# Если AI сломал — мгновенный откат:
git checkout .
# или если уже закоммитил плохое:
git revert HEAD --no-edit
```

**Git-алиас для быстрых чекпоинтов:**
```bash
git config --global alias.cp 'commit -am "checkpoint: pre-AI"'
# Теперь: git cp — и у вас safety net за 1 секунду
```

#### 3. Branch-per-task стратегия
- Для каждой AI-задачи → отдельная ветка
- Имя ветки = scope: `feat/add-password-reset-form`
- Готово и тесты пройдены → merge
- Пошло не так → удалить ветку, main не тронут

#### 4. Scope-контроль
- ВСЕГДА указывай конкретные файлы: `Пофикси @auth-middleware.ts`, а не «пофикси авторизацию»
- Один промпт = одна задача
- Если нужны изменения в >5 файлах → разбей на подзадачи

#### 5. Restore Checkpoint в Cursor
- Cursor имеет встроенную систему чекпоинтов
- После каждого изменения агента можно откатить
- ⚠️ Есть баг: Restore Checkpoint может уничтожить историю изменений (известная проблема на форуме Cursor) — поэтому git-чекпоинты надёжнее

#### 6. Смена модели
Если агент зациклился на одной модели:
- Переключись на другую модель (Claude → GPT или наоборот)
- Другая модель может «увидеть» проблему с другого угла
- Это как спросить совет другого разработчика

## 5. Best Practices вайб-кодинга для разработчиков

### Вайб-кодинг vs Вайб-инженерия

**Вайб-кодинг** (Андрей Карпати, Feb 2025): «Полностью отдаться вайбу, забыть что код существует, не читать диффы»

**Вайб-инженерия** (Саймон Уиллисон, Oct 2025): Использовать AI для ускорения, но сохранять инженерный контроль

**Для опытных разработчиков правильный подход = вайб-инженерия:**
- AI генерирует код → вы ревьюите
- AI предлагает архитектуру → вы принимаете решения
- Роль разработчика: **архитектор и ревьюер**, а не «печатная машинка»

### Как формулировать промпты

**Плохо:**
```
Сделай авторизацию
```

**Хорошо:**
```
Реализуй JWT-аутентификацию для Express API:

1. Middleware в src/middleware/auth.ts
2. Паттерн как в @src/middleware/logging.ts
3. Токен из заголовка Authorization: Bearer
4. Верификация через jsonwebtoken
5. Если невалидный — 401 с JSON-ошибкой
6. Тесты в __tests__/auth.test.ts

Используй наши типы из @src/types/user.ts
```

**Принципы хороших промптов:**
- **Конкретика:** указывай файлы, пути, паттерны
- **Ссылки:** используй @-mentions на существующий код как образец
- **Ограничения:** что НЕ менять, какие файлы НЕ трогать
- **Тесты:** просить тесты = встроенная проверка качества
- **Итеративность:** Describe → Review → Refine (не жди идеала с первого раза)

### Итеративная доработка

```
Шаг 1: "Создай React hook для debounced search input"
Шаг 2: "Добавь TypeScript типы и обработку ошибок"
Шаг 3: "Добавь JSDoc комментарии с описанием debounce-логики"
```

Каждый шаг — маленький, проверяемый, откатываемый.

### Когда НЕ стоит использовать AI

1. **Критические security-компоненты** — шифрование, авторизация, работа с ПДн. AI может пропустить уязвимость. Ревьюьте руками.

2. **Незнакомый domain** — если вы не понимаете сгенерированный код, вы не можете его оценить. Сначала изучите, потом генерируйте.

3. **Сложная бизнес-логика без тестов** — если нет тестов для проверки, AI может «сломать» логику незаметно.

4. **Оптимизация производительности** — AI не профилирует. Он может написать «O(n²) вместо O(n)» и не заметить.

5. **Legacy код без документации** — AI будет галлюцинировать о причинах, почему код написан именно так. Спросите человека.

### Правила для команды (можно адаптировать для Колвир)

1. **Всегда ревьюить AI-код** — как ревью джуниора
2. **Git checkpoint перед каждым промптом** — `git cp`
3. **Максимум 8 сообщений в сессии** — потом новый чат
4. **Один промпт = одна задача** — не мешать всё в кучу
5. **Тесты = обязательная часть промпта** — AI + TDD = лучшая комбинация
6. **.cursor/rules в репозитории** — единые стандарты для всей команды
7. **Ask перед Agent** — сначала исследуй, потом реализуй
8. **Указывай конкретные файлы** — не давай агенту бродить по всему проекту
